# 事实库配置平台 PRD 及设计需求文档
> 基于“事实提取与图谱构建工作台”业务流

---

## 1. 产品概述
### 1.1 背景及目的
随着业务形态向 AI 驱动（尤其是 RAG 架构）演进，基于“语料”的知识库由于结构松散、易造成大模型幻觉，已无法满足高精度问答需求。因此，需要构建一套基于“事实（Fact）”和“实体（Entity）/事件（Event）”图谱的结构化知识库。
本“配置平台”旨在提供一站式的事实提取、实体挂载、冲突校验与入库审核的闭环工作流，降低人工录入成本，提升知识入库的规范性和准确率。

### 1.2 目标用户与使用场景
1. **运营/业务人员**：上传官方文档/白皮书片段，发起知识提取流程，审核确认事实准确性。
2. **算法工程师**：配置文本清洗规则与模型抽取参数。
3. **知识审核员**：处理模型提取出的事实与现有库内知识的冲突（新增、覆盖或剔除），维护知识图谱的健康度。

---

## 2. 核心功能模块划分
平台主要分为四大功能域：
1. **资产大盘（Entities / Events）**：展示和管理全量已知实体与事件。
2. **内容库（Facts Lib）**：管理已入库的结构化事实，支持检索、标签、状态审计。
3. **事实提取（Fact Extraction）**：提供短文本/长文本的双模抽取界面。
4. **日志与诊断**：包括系统审计日志和 RAG/防幻觉诊断。

---

## 3. 详细功能设计需求
### 3.1 事实提取：短文本快捷提取模式
- **定位**：供单次、快速、无复杂业务逻辑的验证或提取。
- **输入交互**：提供文本输入域，支持设定提取模型（如 DeepSeek-v3）。
- **参数控制**：支持切换提取粒度（单选）。
  - **浓缩单一核心事实**（合并为一条核心摘要）。
  - **提取若干分散事实**（拆解为多条独立陈述事实）。
- **输出**：直接输出结构化事实卡片，不进入复杂的确认流。

### 3.2 事实提取：长文本深度解析工作台（重点模块）
- **定位**：面向大批量、结构复杂、易产生图谱映射冲突的真实业务场景。采用“三阶段确认流（Two-stage confirmation）”。

#### 步骤一：解析长文本（上传与规则设定）
1. **上传区**：支持拖拽上传 TXT/CSV/JSON 等文件，或直接粘贴超长文本。
2. **算法预处理与清洗规则设定（复选框区域）**：
   - [ ] 去除无用语气词及口语化表述
   - [ ] 消除代词指代（自动指代消解补全实体名称）
   - [ ] 过滤时效性强且已过期的数据内容
   - [ ] 按句号强制切分长句后逐句提纯事实
   - （支持算法人员自定义勾选，影响后端模型入参）。
3. **提交解析**：点击“开始深度解析”进入模型处理阶段，系统流转至步骤二。

#### 步骤二：确认新实体与事件
1. **业务逻辑**：机器识别文本后，会自动召回库内已知实体，同时会挖掘出“疑似新实体/新事件”。
2. **UI 展示**：将“新挖掘”的实体/事件以卡片列表形式并排展示。
3. **人工判决操作**：
   - “加入新建清单”（默认勾选） -> 同意创建节点。
   - “无效实体/非实体”（打回剔除） -> 拒绝污染图谱。
4. **流转**：点击“确认选中项，并进入事实审核”，生成最终事实映射表，进入步骤三。

#### 步骤三：事实校验与入库（审核判决台）
1. **数据分组逻辑（交通灯法则）**：
   根据步骤二的判决结果与后端知识库的撞库比对，对生成的事实分三大状态流：
   - 🟢 **可直接导入（绿灯）**：事实全新，且挂载的实体/事件均已存在或已在步骤二确认新建。
   - 🟡 **冲突警告（黄灯）**：该事实表述与库内某一已知事实存在语义重合或冲突矛盾。
   - 🔴 **关联缺失（红灯）**：由于用户在步骤二剔除了某些实体，导致当前事实无实体可挂载，成为孤儿事实。
2. **事实卡片 UX/UI 设计（核心体验）**：
   - **比对视图**：卡片内部必须分为“原本的内容（带底色的来源原文）”与“AI 提取的事实”，方便人工迅速核查有无遗漏。
   - **快捷编辑**：提供 `toggleFactEdit` 交互。点击“✏️ 编辑”直接在卡片内将文本展示区转为 `<textarea>` 供人工修正模型错误。
   - **冲突信息提示**：对于黄/红区事实，需高亮展示“🚨 冲突判断原因”以及“现有库内事实”的具体内容，供对比参考。
   - **状态按钮组**：
     - 绿灯区：✅ 确认入库。
     - 黄灯区：提供【强制覆盖】与【作为新事实入库】的分支操作。
     - 红/黄灯区：🗑️ 放弃（直接丢弃该条事实）。
   - **图谱映射态可视化（侧边栏或右侧区块）**：
     需展示 `relations` 关系流：
     - 如果是连接已知实体，显示灰底“关联已知实体”。
     - 如果连接的是步骤二确认的新实体，显示蓝底“✨ 将自动创建新实体”，带有显眼 ✓。
     - 如果连接的是步骤二被剔除的实体，显示红底删除线“已在步骤二剔除”，带有显眼 ✗。

### 3.3 资产大盘（实体与事件图谱管理）
- **功能描述**：用于查看大模型抽取并沉淀的图谱资产。
- **展示形式**：
  - 列表视图：支持条件检索、展示审核状态与来源。
  - 树状视图（Event Tree / Entity Tree）：使用可视化库（如 Vis.js）展示实体间的上下位关系与事件时序关系。

## 4. 关键接口与数据流转结构

### 4.1 关系映射流（Relation Mapping）
为了支持第三步中对“新实体/剔除实体”的状态反馈，后端传递的事实对象结构必须包含详细的 `relations` 数组：
```json
{
  "id": "f_123",
  "content": "全国大赛预赛于3月10日9点开始。",
  "sourceText": "在此次白皮书中提到，各位翘首以盼的全国大赛预赛暂定于3月10日的早晨9点钟拉开帷幕...",
  "status": "yellow",
  "conflictMsg": "与库内事实存在时间/地点重合...",
  "existingFactContent": "全国大赛预赛于3月15日开始。",
  "relations": [
    { "id": "ev_001", "name": "全国大赛预赛", "type": "event", "isNew": false, "skipped": false },
    { "id": "tmp_ent_001", "name": "3月10日", "type": "entity", "isNew": true, "skipped": false },
    { "id": "tmp_ent_002", "name": "早晨", "type": "entity", "isNew": false, "skipped": true }
  ]
}
```
**渲染逻辑**：
前端须拦截对象的 `relations` 字段，解析 `isNew` 和 `skipped` 布尔值，渲染为不同的图谱映射徽标。

### 4.2 交互状态机（State Machine）
操作某张事实卡片后，其前端状态应立即改变，防止重复提交：
- 触发 `accept` / `override`：按钮文本变为“✅ 已成功关联入库”，按钮置灰并 `disabled`，卡片透明度轻微下降。
- 触发 `reject`：卡片整体置灰（`opacity: 0.4`），不可再交互，打上“已放弃”水印。

## 5. 非功能性与体验需求
1. **防误触机制**：进入深度解析第二/三步后，如果在未全部处理完或未保存草稿前点击其他页面，需弹出未保存拦截确认。
2. **返回逻辑**：三阶段流程在步骤二和步骤三底部应提供统一的“返回上一步”按钮，便于返回重新审阅实体或重设清洗规则。
3. **文案一致性**：整个平台需屏蔽容易引起误解的技术词汇（如把 Graph Skeleton 翻译为“骨架”等），全部统一为业务化表述（如“新实体与事件”、“知识映射”）。
