# 事实提取与图谱构建工作台 - 界面交互设计需求文档 (V1)

## 一、 项目背景与目标

### 业务背景
*   **行业痛点**：当前基于大模型的游戏客服与助手，往往依赖松散的非结构化语料（如原样抛入整篇长图文攻略），极易导致模型产生“幻觉”或胡编乱造，且知识更新滞后。
*   **团队目标**：为了提升 LLM 回答问题的绝对准确性，我们需要将松散的业务文档转化为高质量的“结构化事实（Fact）”，并将其挂载到特定的游戏“实体（Entity）”和“事件（Event）”图谱上。

### 核心目标与定位
*   **定位**：打造一个“一站式的人机协同知识构建配置工作台”。
*   **受众**：业务运营同学（提交需求文档）、算法工程师（配置规则）、知识审核员（审核图谱与事实冲突）。
*   **核心任务**：
    *   **降本增效**：通过引入“预处理清洗”与“大模型提取”的前置工作流，将纯人力的逐条录入转变为“机器粗排 + 人工判决”。
    *   **质量保障**：提供直观的“冲突对比”与“知识映射可视化”，保证入库的事实 100% 准确且无逻辑断层。

## 二、 整体设计风格与交互原则

### 2.1 整体设计风格（视觉开动前参考）
*   **色调**：
    *   **主色调**：科技蓝（`#165DFF`），传达专业与严谨，用于主操作按钮与高亮标识。
    *   **状态色系（极简交通灯）**：
        *   🟢 成功绿（`#00B42A`）：用于无冲突可直接入库的状态。
        *   🟡 警告黄（`#F7BA1E`）：用于与库内事实冲突、需人工覆盖的状态。
        *   🔴 危险红（`#F53F3F`）：用于实体缺失、严重错误或放弃操作的状态。
*   **背景风格**：干净的灰白底色（`#F2F3F5` / `#FFFFFF`），利用卡片阴影（`Box-shadow`）做层级划分，避免花哨干扰审核员视线。

### 2.2 交互原则
*   **沉浸式工作流**：长文本深度解析采用“三阶段确认流”的沉浸式体验，步骤间平滑过渡，核心操作区保持在视口中心。
*   **同屏比对**：在事实审核环节，坚持“原文”与“AI提取内容”在同一卡片内左右或上下同屏展示，减少用户的视线跳转。
*   **操作即刻反馈**：任何入库或放弃操作，卡片必须即刻响应（如变灰、按钮变为已处理），避免重复点击。

## 三、 页面详细逻辑与界面交互梳理

以下将按“工作台”的核心模块分屏/分步骤进行详细设计梳理。

> **思路**：提供快速切换功能入口以及整体数据的宏观视角。

*   **1. 左侧导航栏**
    *   **结构**：采用可折叠的树状菜单。
        *   图谱资产（包含：实体库、事件树）
        *   知识录入（包含：事实提取工作台）—— **[Default 默认高亮焦点]**
        *   内容管理（包含：事实管理库）
    *   `[占位图：左侧深色/浅色侧边导航栏UI截图]`
*   **2. 资产大盘页 (Entities / Events)**
    *   **内容**：顶部是四个核心数据卡片（实体总数、事件总数、本周新增事实等）。下方是支持多条件检索（类型、标签、时间）的表格清单。
    *   **交互**：点击表格行，右侧平滑滑出抽屉（Drawer），展示该实体的详细属性和绑定的下游事实。

### 第二模块：事实提取 - 短文本快捷提取
> **思路**：简单直接，用于日常少量知识点的快速测试与单点提取，无需繁琐的审核流程。

*   **页面结构**：
    *   **提取模式切换器**：放置在页面最上方，两个大按钮供切换“短文本快捷提取”与“长文本深度解析”。
    *   **输入区**：大面积纯净的 Textarea 文本框，带有必填星号。
    *   **配置项区**：
        *   单选 Radio：提取粒度（浓缩单一核心事实 vs 提取若干分散事实）。
        *   下拉选框/输入框：指定提取模型。
    *   **操作与结果反馈**：
        *   点击“提取事实”按钮后，按钮转圈 Loading，下方出现骨架屏等待效果。
        *   提取成功后，直接在下方纵向罗列“事实卡片”（无需红黄绿分组）。
    *   `[占位图：短文本快捷提取模式完整UI设计稿]`

### 第三模块：事实提取 - 长文本深度解析工作台 (核心业务流)
> **思路**：这是解决实际复杂长文本的重头戏，采用明确的“步骤条指示器”，配合算法预处理与人工判决，达成极高准确率的入库目的。

#### 第 1 步：解析长文本 (预处理与配置)
*   **场景说明**：业务运营拿到长篇资料后，第一步需要设定规则让机器去“粗筛”。
*   **界面模块**：
    *   **进度条指示器**：悬浮于顶部（📝 步骤一：解析长文本 -> 🔍 步骤二 -> ✅ 步骤三）。当前步骤一呈科技蓝色高亮。
    *   **文件上传拖拽区**：支持点击上传或将文件拖入虚线框内（大号 Icon 提示支持 txt, json 等格式）。
    *   **纯文本输入区（备用）**：与拖拽区并列或提供 Tab 切换。
    *   **算法预处理规则池（复选框矩阵）**：
        *   [ ] 消除代词指代补全
        *   [ ] 过滤过期时效性表述
        *   [ ] 去除口语化及无关人设
    *   **交互动作**：点击【开始深度解析】大按钮，进入全局 Loading 状态，流转至步骤二。
    *   `[占位图：长文本提取步骤一_包含拖拽上传与规则勾选区域的UI截图]`

#### 第 2 步：确认新实体与事件 (图谱防抖拦截)
*   **场景说明**：在大量事实生成前，先把“疑似新挖掘”的词根揪出来让人确认，防止错别字或脏数据污染核心图谱。
*   **界面模块**：
    *   **顶部提示**：告知用户“AI从本次文本中发现了以下潜在的新实体/事件，请确认是否允许创建”。
    *   **卡片网格阵列 (Grid Layout)**：将所有新实体平铺展示。
        *   单张卡片包含：`实体名`、`推断类型徽标`、`右侧勾选/剔除开关`。
        *   卡片底部提供：`[下拉展开图标] 查看推断出处原文片段`。
    *   **操作栏**：提供“全选”与“批量剔除”快捷键。
    *   **交互动作**：点击底部【确认选中项，并进入事实审核】，系统将这些实体的状态打入缓存，流转至终审。
    *   `[占位图：步骤二_新实体网格展示与判决开关UI截图]`

#### 第 3 步：事实校验与入库 (冲突裁决与同屏比对)
*   **场景说明**：这是审核员耗时最长的环节。系统需要把 AI 提取的事实与现有数据库做对比，按“交通灯”原则分类，审核员对症下药。
*   **界面结构布局**：
    *   **顶部 Sticky 导航签**：
        *   Tab 1：🟢 绿灯区（可直接导入）（附带数量 Badge）
        *   Tab 2：🟡 黄灯区（冲突警告）（附带数量 Badge，红色强提示）
        *   Tab 3：🔴 红灯区（关联缺失）（附带数量 Badge）
    *   **核心组件：事实同屏比对卡片 (Fact Card)**
        *   **卡片头部 (Header)**：展示左侧状态徽标（如 `⚠️ 冲突警告`），右侧操作按钮组（`✅ 确认入库` / `强制覆盖` / `✏️ 编辑` / `🗑️ 放弃`）。
        *   **卡片主体左侧 (信息比对区)**：
            *   上部：灰色圆角底色的**原本的内容（来源片段）**。
            *   中部：大号字体的**AI 提取的事实**（如进入编辑态，此区域原地翻转/转换为高亮边框的 Textarea 输入框）。
            *   下部（仅黄红灯可见）：带有警示色的边框框，展示**🚨 冲突判断原因**与对比的**现有库内事实**。
        *   **卡片主体右侧 (图谱知识映射状态)**：
            *   列表化展示该事实关联的所有实体。
            *   用不同的背景色和图标表达映射状态：
                *   灰底/✓：关联已知实体。
                *   蓝底/✓：✨ 将自动创建新实体（对应步骤二勾选项）。
                *   红底删除线/✗：已在步骤二剔除（导致红灯报错原因）。
    *   **底部动作条**：悬浮底栏（Fixed Bottom），提供“一键导入当前页无冲突事实”以及“返回上步重新审阅”的全局兜底操作。
    *   `[占位图：步骤三_黄灯区带有冲突比对面板与右侧知识映射关系的卡片细节UI截图]`
    *   `[占位图：步骤三_卡片进入内联编辑模式时的UI状态截图]`

## 四、 异常情况交互展示 (Error Handling UI)

*   **大模型服务超时兜底页**：
    *   如果在步骤一点击解析后长达 60s 无响应， Loading 动画中断，弹出居中弹窗：展示超时插画，并提示“由于文本体量过大或模型网络波动解析超时”，提供【重新尝试】与【切换为短文本拆分提取】两个出口。
    *   `[占位图：模型超时兜底弹窗设计稿]`
*   **离开当前工作台挽留拦截**：
    *   在进入长文本第二/三步工作流后，若用户误触浏览器的“后退”或点击了左侧菜单的其他页面，屏幕中央必须弹出二次确认 Modal：“当前提取任务存在未保存的更改，确认要强行离开并丢弃进度吗？”。
    *   `[占位图：防误触离开的二次确认弹窗设计稿]`
