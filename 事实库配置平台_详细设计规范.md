# 事实库配置平台 - 详细交互与设计需求说明书 (V2 细化版)

本篇文档在上一版 PRD 的基础上，针对系统的**界面交互规范、数据流协议、异常兜底机制以及角色权限**进行了全面细化，旨在作为前端 UI 开发、交互设计和后端 API 研发的直接参考标准。

---

## 一、 角色与权限模型 (RBAC)

平台根据实际业务工作流，划分三个核心角色：

| 角色 | 权限范围 | 典型操作 |
|---|---|---|
| **业务运营** | 基础使用权 | 上传业务文档，发起提取任务；查看并检索当前库内的各类图谱资产。 |
| **数据审核员** | 审核操作权 | 拥有工作台第二、第三步的处理权限。可处理冲突事实，审批/驳回新实体的创建，修改事实文本。 |
| **算法/超管** | 系统配置权 | 调整大模型提取参数（Temperature/Top-P）、配置并维护底层的“预处理与清洗规则库”，查看报错诊断日志。 |

---

## 二、 全局 UI/UX 规范与组件定义

为了确保配置平台的体验统一，全面采用类似 Arco Design / Ant Design 的现代企业级设计规范。

### 2.1 品牌色与状态色 (Color Tokens)
*   **主色调 (Primary)**：`#165DFF` (科技蓝) —— 用于所有主要的“下一步”、“提交”、“确认入库”按钮。
*   **成功态 (Success/Green)**：`#00B42A` —— 绿灯区无冲突事实、已成功入库状态。
*   **警告态 (Warning/Yellow)**：`#F7BA1E` —— 黄灯区库内冲突、强覆盖警告。
*   **危险态 (Danger/Red)**：`#F53F3F` —— 红灯区严重缺失、放弃/删除按钮、拦截报错。
*   **中性色/禁用态 (Neutral)**：`#E5E6EB` (边框)、`#F2F3F5` (浅色背景区)。

### 2.2 核心交互组件行为
*   **按钮交互**：
    *   Hover: 亮度增加 / 透明度变为 0.8。
    *   Click (Loading): 发起网络请求时，主按钮显示 Loading 旋转图标，禁用页面其他提交按钮。
    *   Disabled: 置灰 (`opacity: 0.5`, `cursor: not-allowed`)。
*   **局部抽屉/弹窗**：所有的树形图谱展示、编辑实体属性等次级操作，均使用侧边抽屉（Drawer）或模态框（Modal）承载，避免打断当前长任务的工作流。

## 三、 长文本深度解析工作台 - 核心模块详细设计

### 3.1 步骤一：解析长文本 (File & Rules)
**【输入规范】**
*   **文件上传**：支持 `.txt`, `.csv`, `.json`, `.docx`。限制单文件大小不超过 **10MB**。
*   **文本输入框**：支持直接粘贴。字符数限制：**最大 100,000 字符**。超出时在输入框右下角显示红色超限提示，并禁用“开始解析”按钮。
*   **预处理规则库**：
    *   需提供下拉记忆功能：记住该用户上一次勾选的规则组合。

**【交互反馈】**
*   点击提交后，进入 **全屏遮罩 + 进度条 Loading** 状态。文案轮播提示：“正在进行文本清洗 (20%)” -> “大模型节点抽取中 (60%)” -> “生成图谱映射关系 (90%)”。

### 3.2 步骤二：确认新实体与事件 (Entity Validation)
**【数据加载与展示】**
*   **分页/滚动**：若大模型挖掘出的新实体超过 50 个，采用**虚拟列表 (Virtual List)** 或**触底加载**，保证页面不卡顿。
*   **卡片字段**：每个新实体卡片需展示 `[实体名称]`, `[推断类型（如：人名/地名/业务专有名词）]`, `[出处原文片段]`，帮助审核员快速判断。
*   **批量操作**：提供“全选加入清单”、“全选剔除”的顶部操作栏。

**【临界状态管理】**
*   如果用户将某个实体状态从“加入清单”切换为“剔除”，前端需在内存中更新映射树，准备在第三步将挂载了该实体的事实标记为“红灯”。

### 3.3 步骤三：事实校验与入库 (Conflict Resolution Dashboard)
这是整个系统交互最密集的区域，需实现**极致的键盘+鼠标操作流**。

**【视觉层级与分组】**
*   使用 Sticky 导航栏固定在页面顶部，包含绿灯/黄灯/红灯三个 Tab，并显示徽标数字（如 `绿灯区 42` | `黄灯区 5` | `红灯区 1`）。
*   默认优先定位到“黄灯区”，因为这里最需要人工介入。

**【卡片级详细交互 (Fact Card)】**
1.  **卡片状态转化 (State Machine)**：
    *   初始态：显示编辑、确认、放弃等操作按钮。
    *   编辑态 (`toggleFactEdit`)：点击“编辑”，卡片事实展示区平滑展开为 `textarea`。提供“保存”与“取消”按钮。
    *   已决态 (Resolved)：执行任何入库/放弃操作后，卡片立即增加一层白色半透明遮罩 (`rgba(255,255,255,0.6)`)，操作按钮变更为“✅ 已处理”并禁用，且不可撤销（除非依靠右上角的全局回滚）。
2.  **右侧侧边栏（图谱映射可视化）**：
    *   以微型连线图或条目列表形式，直观表明：`[当前事实] ——属于——> [事件A (已知)]`、`[当前事实] ——包含——> [实体B (新造，蓝色高亮)]`。
    *   若实体被判定缺失，对应的连线显示为红色断裂状态。

**【全局动作兜底】**
*   在页面底部提供**“一键导入当前 Tab 全部无冲突事实”**按钮，并要求二次弹窗确认（“确定要批量导入 42 条事实吗？”）。
*   离开当前路由拦截：如果还有未处理的“黄/红灯”事实，用户点击左侧菜单切换页面时，必须弹窗拦截：“当前工作流存在未保存的变更，确定要离开吗？”。

## 四、 核心数据模型 (Data Schema)

为了支撑前端展现，后端 API（尤其是第三步获取事实列表的接口）需严格遵守以下 JSON Schema：

### 4.1 事实对象定义 (Fact Schema)
```json
{
  "fact_id": "string (唯一标识)",
  "content": "string (AI抽取的提纯文本)",
  "source_text": "string (对应的原始长文本片段)",
  "status": "enum('green', 'yellow', 'red')",
  "conflict_info": {
    "has_conflict": "boolean",
    "conflict_reason": "string (如：时间节点矛盾)",
    "existing_fact_id": "string (如果与库内事实冲突，提供库内ID)",
    "existing_fact_content": "string (库内原文，供前台比对)"
  },
  "relations": [
    {
      "target_id": "string (实体或事件的临时ID或真实ID)",
      "target_name": "string",
      "target_type": "enum('entity', 'event')",
      "is_new": "boolean (是否为本次大模型新挖掘的)",
      "action": "enum('keep', 'discard') (对应步骤二用户的操作结果)"
    }
  ]
}
```

## 五、 异常场景与兜底机制

1.  **大模型超时 / 宕机**：
    *   若抽取请求超过 60 秒未响应，前端自动中止，并弹窗提示：“模型分析超时，可能是文本过于庞大，建议截断后分批提交”。
    *   同时提供“降级为纯人工录入模式”的快捷入口。
2.  **并发操作冲突（乐观锁机制）**：
    *   如果两个审核员同时处理同一批语料：后端利用版本号（Version）或时间戳控制。若保存时库内实体发生改变，提示：“部分库内事实已被他人修改，请刷新比对最新状态”。
3.  **JSON 格式解析崩溃**：
    *   由于大模型输出幻觉导致返回的 JSON 结构损坏时，后台拦截解析错误，并向前端返回 `500 - AI 返回格式非法`，避免前端渲染白屏。前端捕获后提示：“本次提取失败（AI结构损坏），请重试”。
4.  **长文本输入性能**：
    *   对于过长的复制粘贴事件，输入框应进行防抖（Debounce）处理，防止浏览器渲染主线程阻塞。

## 六、 审计与追溯要求

*   **操作日志 (Audit Log)**：针对每一次事实卡片的“强制覆盖”、“放弃”以及“修改并入库”，系统必须默默记录操作流水（`Operator_ID`, `Timestamp`, `Action`, `Before_Value`, `After_Value`）。
*   在“资产大盘 - 内容库”中，每一条事实均需展示 **[来源：AI提取/人工创建]** 以及 **[最后修改人]** 标签，实现知识溯源。
